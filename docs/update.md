# Оновлення системи (OTA sysupgrade)

Цей документ описує процес оновлення прошивки на вузлах меш-мережі, що вже знаходяться у роботі, без фізичного доступу до кожного роутера.

> **Відмінність від bulk-flasher:**
> `flasher.py` призначений для **першої** прошивки роутерів через TFTP — коли OpenWrt ще не встановлено.
> `ota_upgrade.py` призначений для **оновлення** вузлів, що вже запущені у мережі, через SSH.

---

## Як це працює

Скрипт `bulk-flasher/ota_upgrade.py` виконує наступну послідовність для кожного вузла:

1. Відновлює IP-адресу вузла з `topology.json` (той самий детермінований алгоритм, що і в `generate_configs.py`).
2. Підключається до вузла по **SSH** з використанням ключа `configs/keys/<NODE>_id_rsa`.
3. Завантажує відповідний `*sysupgrade*.bin` образ через **SCP** у `/tmp/sysupgrade.bin` на роутері.
4. Запускає `sysupgrade -n /tmp/sysupgrade.bin` на роутері — роутер автоматично перезавантажується з новою прошивкою.

## Передумови

### 1. Залежності Python

```bash
pip install paramiko scp
```

### 2. Наявність необхідних файлів

| Файл | Де має бути | Як отримати |
|------|-------------|-------------|
| `*sysupgrade*.bin` | `openwrt-generator/bin/<NODE>/` | `python3 generate_configs.py --build` |
| `<NODE>_id_rsa` | `openwrt-generator/configs/keys/` | генерується автоматично при `generate_configs.py` |
| `topology.json` | `openwrt-generator/topology.json` | вхідний файл топології |

### 3. Мережева доступність

Машина оператора має бути **в одній підмережі** з вузлами меш-мережі (за замовчуванням `10.0.0.0/24`), або мати маршрут до неї.

Перевірка перед запуском:
```bash
# Перевірити доступність конкретного вузла (наприклад, A1 = 10.0.0.1)
ping -c 3 10.0.0.1

# Перевірити SSH-доступ вручну
ssh -i openwrt-generator/configs/keys/A1_id_rsa root@10.0.0.1
```

## Використання

### Базовий запуск — оновити всі вузли

```bash
cd mesh-project
python3 bulk-flasher/ota_upgrade.py
```

Скрипт виведе список вузлів, образів та IP-адрес і запитає підтвердження перед початком.

### Вибіркове оновлення конкретних вузлів

```bash
python3 bulk-flasher/ota_upgrade.py --nodes A1 B1 C2
```

### Паралельне оновлення всіх вузлів одночасно

```bash
python3 bulk-flasher/ota_upgrade.py --parallel
```

> ⚠️ Паралельний режим прискорює процес, але роутери перезавантажуються одночасно — mesh-мережа тимчасово повністю недоступна.

### Перегляд без виконання (dry-run)

```bash
python3 bulk-flasher/ota_upgrade.py --dry-run
```

Виводить усі команди та перевірки, але нічого не виконує реально. Корисно для перевірки конфігурації перед оновленням.

## Повний список аргументів

| Аргумент | За замовчуванням | Опис |
|----------|-----------------|------|
| `--topology` | `../openwrt-generator/topology.json` | Шлях до файлу топології |
| `--bin-dir` | `../openwrt-generator/bin` | Папка з папками вузлів та `*sysupgrade*.bin` |
| `--keys-dir` | `../openwrt-generator/configs/keys` | Папка з SSH-ключами `<NODE>_id_rsa` |
| `--subnet` | `10.0.0.` | Префікс IP-підмережі |
| `--user` | `root` | Ім'я користувача SSH |
| `--nodes` | всі | Перелік вузлів для оновлення |
| `--parallel` | вимкнено | Паралельна прошивка всіх вузлів |
| `--dry-run` | вимкнено | Тільки показати дії без виконання |

## Процес оновлення крок за кроком

### Крок 1: Зібрати нові образи

Якщо оновлення передбачає зміну конфігурації або пакетів — перегенеруйте конфіги та зберіть образи:

```bash
cd openwrt-generator

# Відредагуйте topology.json або generate_configs.py за потреби
# Потім:
python3 generate_configs.py --build
```

Якщо топологія не змінилась, а потрібно тільки оновити версію OpenWrt — достатньо перезібрати ImageBuilder.

### Крок 2: Перевірка доступності мережі

```bash
# Перевірити, що всі вузли доступні
for ip in 10.0.0.1 10.0.0.2 10.0.0.3; do
    ping -c 1 -W 1 $ip && echo "$ip — OK" || echo "$ip — НЕДОСТУПНИЙ"
done
```

### Крок 3: Dry-run перевірка

```bash
python3 bulk-flasher/ota_upgrade.py --dry-run
```

Переконайтеся, що для кожного вузла знайдено образ і ключ.

### Крок 4: Запуск оновлення

```bash
# Послідовно (безпечніше — мережа деградує поступово)
python3 bulk-flasher/ota_upgrade.py

# або паралельно (швидше, але мережа падає повністю)
python3 bulk-flasher/ota_upgrade.py --parallel
```

### Крок 5: Очікування завершення

Після запуску `sysupgrade` роутер **перезавантажується ~90–120 секунд**.
Скрипт нагадає про це після завершення роботи.

### Крок 6: Перевірка після оновлення

```bash
# Перевірити версію прошивки на вузлі
ssh -i openwrt-generator/configs/keys/A1_id_rsa root@10.0.0.1 "cat /etc/openwrt_release"

# Перевірити batman-adv
ssh -i openwrt-generator/configs/keys/A1_id_rsa root@10.0.0.1 "batctl meshif bat0 n"
```

## Типові помилки та їх вирішення

| Помилка | Причина | Вирішення |
|---------|---------|-----------|
| `Authentication failed` | Ключ не відповідає `authorized_keys` на роутері | Переконайтесь, що ключі з `configs/keys/` збігаються з тим, що було записано під час генерації |
| `Connection timed out` | Вузол недоступний по мережі | Перевірте ping, batman-adv, фізичне підключення |
| `No sysupgrade*.bin found` | Образ не зібраний | Запустіть `generate_configs.py --build` |
| `No space left on device` | Замало місця у `/tmp` | Підключіться вручну і очистіть `/tmp` |
| SSH-з'єднання обривається під час SCP | Роутер перезавантажується або нестабільний | Можливо, роутер ще не завантажився після попередньої прошивки — зачекайте |

## Відмінність `-n` від стандартного sysupgrade

Скрипт використовує `sysupgrade -n`, що означає **скидання конфігурації до заводської**.
Це зроблено навмисно: вся конфігурація вузла вбудована в образ під час генерації (`generate_configs.py`), тому збереження старих налаштувань непотрібне і може викликати конфлікти.

Якщо потрібно зберегти конфігурацію (`sysupgrade` без `-n`), відредагуйте рядок у `ota_upgrade.py`:
```python
# Поточний варіант (скидає конфіг):
"sysupgrade -n /tmp/sysupgrade.bin"

# Варіант зі збереженням конфігурації:
"sysupgrade /tmp/sysupgrade.bin"
```
